# Internet of Dingus Ryan Parsley Notes: I hope to demistify the internet of things by creating an example to illustrate all the parts that make it work.
> The Internet of Things (IoT) is the network of physical devices, vehicles, home appliances and other items embedded with electronics, software, sensors, actuators, and connectivity which enables these objects to connect and exchange data. Each thing is uniquely identifiable through its embedded computing system but is able to inter-operate within the existing Internet infrastructure. —[Wikipedia](https://en.wikipedia.org/wiki/Internet_of_things) Notes: Go ahead, soak that up. So, a thing can be a device, or a car, or an apliance or... well... anything. Maybe the next line clears it up. Sensor (input), acutators (output) and connectivity (internet). Uniquely Identifiable Able to inter-operate withing the existing internet infrastructure.
* Sensors (Input) * Acutators (output) * Uniquely Identifiable * Connectivity (internet). * Able to inter-operate withing the existing internet infrastructure. (yes, *that* internet) Note: * Sensors can be buttons or components that sense thinks like heat, humidity, light, proximity * Acutators can be thought of as components that create output. Motors, Speakers, lights, relays. * Uniquely Identifiable, we need to know which thing we're talking too or hearing from. * The last two go together. The "thing" needs to be on the internet and not in too novel of a way.
# Enter the dingus Note: A dingus is... a kind of a thing. For the rest of this talk, when I say "dingus" I'm talking about this hardware profile.
## A Dingus ![A Dingus](img/dingus.png) Note: A dingus has: * Wifi * One actuator * One sensor The reference dingus is a Huzzah Feather from a company called Adafruit. It's compatible with Arduino, but has wifi built in.

## Internet of Dingus ![Internet of Dingus](img/internetOfDingus.jpg) Note: 3 servers come together to provide a specific piece of the puzzle. Each dingus only interacts with one of the servers directly. You don't *need* this level of separation, but I think it helps distil the conversation.

## WTF is MQTT!? The de facto way to have things talk amongst themsleves. * Publisher(s) * Subscriber(s) * Broker Note: It's a standard way of enabling publishing and subscribing messages. If you think about MQTT as chat for things, the broker is the buddy list.
## The Frontend ![Frontend](img/Internet_of_Dingus_view.png) Note: The frontend ties together realtime data from MQTT to the less dynamic meta data served via API that Drupal is responsible.
